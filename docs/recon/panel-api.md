# Panel API REST surface (source-mined from ReFxHosting/apps/panel-api)

> Recon agent output, 2026-07-13. Source-grounded; file paths cited are relative to the named repo.

## Summary

The panel-api (NestJS 11, Express, port 4000, global prefix `api/v1`) exposes a customer REST surface guarded by JwtAuthGuard (Bearer access-JWT) + per-server PermissionGuard using permission strings from packages/shared/src/permissions.ts. All successful responses are wrapped `{ success: true, data }` (paginated: `{ success, data, meta:{page,pageSize,total,totalPages} }`) by a global TransformInterceptor; errors are `{ statusCode, error, message, path, timestamp }`. Users can mint long-lived scoped API keys (scopes READ/WRITE/ADMIN/STATUS_READ, format `refx_<8-char-prefix><secret>`) that authenticate ONLY via the `X-Api-Key` header on normal endpoints â€” a refx_ key presented as `Authorization: Bearer` fails JWT verification everywhere except `GET /api/v1/status/nodes` (STATUS_READ tokens accepted as Bearer there). Live console is a Socket.IO gateway at namespace `/ws/console` authenticated with the ordinary access JWT in the handshake (`auth.token`) â€” there is NO dedicated console-token REST endpoint. Full endpoint inventory extracted for auth/2FA, account, API keys, servers, power, command, files, backups, databases, schedules, sub-users, allocations, startup/variables, switch-game, upgrade/resize, mods/modpacks/workshop/voice, SFTP, stats, plus verbatim e2e-asserted bodies. Notable quirks found: no `PATCH /servers/:id` rename endpoint exists even though apps/web calls it; the WS `command` event checks the non-catalog string `control.console` (sub-users granted `console.command` can't use WS commands, only REST); duplicate sub-user routes registered by two controllers with different permission strings.

# 1. Global conventions

**Response envelope** â€” `apps/panel-api/src/common/interceptors/transform.interceptor.ts`: every successful REST response is wrapped `{ success: true, data: <payload> }`. If the handler's return value already contains a `meta` key (paginated), it is spread instead: `{ success: true, data: [...], meta: {...} }`. `null`/`undefined` returns pass through unwrapped (204s have no body). Routes marked `@RawResponse()` (only `GET /metrics`) skip the envelope.

**Pagination** â€” `apps/panel-api/src/common/dto/pagination.dto.ts`: query params `page` (default 1, min 1), `pageSize` (default 25, min 1, max 100), optional `q` free-text filter. Response meta: `{ page, pageSize, total, totalPages }` via `paginate()` helper.

**Error shape** â€” `apps/panel-api/src/common/filters/all-exceptions.filter.ts` (registered globally in main.ts:159): `{ statusCode: number, error: string, message: string | string[], path: string, timestamp: ISO-8601 }`. `error` is e.g. `"NotFoundException"`, `"CONFLICT"`, `"NOT_FOUND"`, `"BAD_REQUEST"`, `"DB_ERROR"`, `"INTERNAL_ERROR"`. Validation errors give `message` as a string array (asserted in `apps/panel-api/test/auth.e2e-spec.ts:82-83`). ValidationPipe is `whitelist: true, forbidNonWhitelisted: true` (main.ts:150-157) â€” unknown body fields â†’ 400.

**Rate limiting** â€” `@nestjs/throttler` ^6.5.0 with Redis storage, registered as global `APP_GUARD` (`apps/panel-api/src/app.module.ts:83-101,157`). Global default: 120 requests / 60 s per IP (`THROTTLE_TTL` default 60, `THROTTLE_LIMIT` default 120 â€” configuration.ts:181-184). Route-specific tighter limits via `@Throttle`: login 10/min, `mfa/verify` 5/min, webauthn login options/verify 5/min, register 5/min, resend-verification 3/min (auth.controller.ts), `GET /status/nodes` 30/min per token (status.controller.ts:54), `GET /tools/minecraft-status` 10/min (tools.controller.ts:17). `TRUST_PROXY` defaults to 1 hop so per-IP limits use X-Forwarded-For (main.ts:93-99). Rate-limit response headers are whatever @nestjs/throttler v6 emits by default â€” nothing custom in the repo.

**Versioning** â€” the version lives only in the `api/v1` prefix; Nest URI versioning is deliberately NOT enabled (main.ts:144-148 comment). `health`, `metrics`, `graphql`, `docs` are excluded from the prefix (main.ts:141-143): `GET /health` returns (envelope-wrapped) `{ status: 'ok', uptime, timestamp, checks: { database: 'up'|'down' } }` (`apps/panel-api/src/platform/health.controller.ts`); `GET /metrics` is raw Prometheus text (`apps/panel-api/src/platform/metrics.controller.ts`). Test asserts `/api/v1/health` is 404 (`apps/panel-api/test/health.e2e-spec.ts`). Swagger at `/docs` only when non-prod or `ENABLE_API_DOCS=true` (main.ts:169-196). GraphQL at `/graphql` (read-only queries; playground/introspection disabled in prod).

**mustChangePassword lock** â€” global `PasswordChangeInterceptor` (`apps/panel-api/src/common/interceptors/password-change.interceptor.ts`): when the principal has `mustChangePassword`, every authenticated route 403s with body `{ statusCode: 403, code: 'PASSWORD_CHANGE_REQUIRED', message: 'Password change required' }` except routes marked `@AllowWhenPasswordExpired()` (`GET /auth/me`, `POST /auth/refresh`, `POST /auth/logout`, `POST /account/password`).

**Request body limits** â€” JSON bodies capped at 5 MB; `POST /api/v1/servers/:id/files/upload` uses a raw binary parser capped at 40 MB (agent enforces 32 MiB) (main.ts:107-127). HTTP timeouts: requestTimeout 240 s, headersTimeout 60 s, keepAliveTimeout 65 s (main.ts:212-215).

# 3. API keys â€” definitive answers

- Users mint scoped keys at `POST /api/v1/auth/api-keys` OR `POST /api/v1/account/api-keys`. The two mint endpoints return DIFFERENT field names for the one-time plaintext: `/auth/api-keys` â†’ `key`; `/account/api-keys` â†’ `token` (plus `{ id, name, prefix, scopes, expiresAt }`) (account.controller.ts:92-115). List: `GET /api/v1/account/api-keys` â†’ array of `{ id, name, prefix, scopes, allowedIps, lastUsedAt, expiresAt, revokedAt, createdAt }` (api-key.service.ts:53-69). Revoke: `DELETE /api/v1/account/api-keys/:id` â†’ 204 (soft revoke).
- **Scopes** are the Prisma enum `ApiKeyScope` = `READ | WRITE | ADMIN | STATUS_READ` (`database/prisma/schema.prisma:252-262`) â€” NOT the per-server permission strings.
- **Key format**: `refx_<prefix(8)><secret>`; stored as SHA-256 hash; expiry, revocation, and per-key CIDR allowlist enforced in `apps/panel-api/src/auth/api-key.service.ts:81-129`.
- **How a key authenticates**: `JwtAuthGuard.canActivate` (`apps/panel-api/src/auth/guards/jwt-auth.guard.ts:34-52`) checks the `X-Api-Key` request header FIRST; if present it authenticates via ApiKeyService and attaches an AuthUser with `apiKeyId` + `apiKeyScopes`. Otherwise it falls back to passport-jwt Bearer verification. **A refx_ key sent as `Authorization: Bearer` is NOT accepted on normal endpoints** (it fails JWT signature verification). The single exception: `GET /api/v1/status/nodes` is @Public + `StatusReadGuard`, which accepts `Authorization: Bearer <refx_ token>` or `X-Api-Key` and requires the `STATUS_READ` scope (`apps/panel-api/src/status/status-read.guard.ts:18-58`).
- **Scope ceiling enforcement**: an API-key principal inherits the owner's RBAC permissions, but (a) global `ApiKeyWriteScopeInterceptor` (`apps/panel-api/src/common/interceptors/api-key-write-scope.interceptor.ts`, registered app.module.ts:167) blocks POST/PUT/PATCH/DELETE unless scopes include WRITE or ADMIN, on every controller; also blocks STATUS_READ-only keys from any JwtAuthGuard route (403 "This token is scoped to the status feed only"); (b) PermissionGuard repeats the write-scope check on per-server routes (permission.guard.ts:78-83). GET/HEAD/OPTIONS are allowed for any scope. So a READ key can drive every GET a session could, including files `contents`/`download-url`.

# 11. Live console WebSocket â€” `apps/panel-api/src/agent/console.gateway.ts`

- The web client also emits a `power` event (`apps/web/lib/ws.ts:119`) but the gateway defines no `power` @SubscribeMessage handler â€” power must go through REST.
