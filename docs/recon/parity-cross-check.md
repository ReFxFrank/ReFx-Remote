# Adversarial cross-check: panel-api vs Android client

> Recon agent output, 2026-07-13. Source-grounded; file paths cited are relative to the named repo.

## Summary

Both reports are accurate about their own repos on the three critical contracts (envelope success shape, auth/refresh flow, console handshake â€” all source-confirmed verbatim), but cross-checking them against source exposed four genuine client-vs-server incompatibilities the Android app ships with and a desktop client must not copy: (1) the MFA login branch is broken â€” the panel returns empty-string tokens with `mfaRequired: true` while Android detects MFA by null tokens, so MFA users cannot log in; (2) Android updates variables via `PATCH servers/{id}/variables` `{envName,value}` but the panel only defines `PUT servers/:serverId/variables/:envName` `{value}` â€” 404; (3) Android decodes the API-key mint at `POST account/api-keys` expecting a required `key` field but that endpoint returns the plaintext as `token` (only `/auth/api-keys` returns `key`) â€” decode failure; (4) Android hands the panel's relative signed download URLs to an https-only opener, so file downloads always fail. Additionally, the `{success:false,error:{message,code}}` error envelope the Android client parses is never emitted by this backend â€” the real error body is the flat `{statusCode, error, message, path, timestamp}` shape the panel report describes.

## Key facts

- `PATCH support/tickets/{id}` and `POST support/tickets/{id}/assign` exist but are AdminPermissionGuard + `support.manage` staff-only (support.controller.ts:83-85,125-127) â€” Android's SupportApi lists them without the caveat.

- Duplicate sub-user route sets confirmed verbatim: subuser.* strings in users.controller.ts:96-143 vs user.* strings in servers.controller.ts:852-881; UsersModule imported before ServersModule (app.module.ts:134,137) â€” route-collision winner still needs runtime confirmation.

## (a)+(b) Discrepancies, with verdicts

### A. Genuine cross-repo incompatibilities (Android client vs panel-api â€” a desktop client must NOT copy these)

**A1. MFA login is broken end-to-end â€” the reports' facts collide and neither flagged it.**
- Panel (verified, `apps/panel-api/src/auth/auth.service.ts:246-253`): the MFA challenge response is `{ accessToken: "", refreshToken: "", expiresIn: 0, mfaRequired: true, mfaToken, methods }` â€” **empty strings, not nulls**.
- Android (verified, `app/src/main/java/gg/refx/android/data/model/Auth.kt:38-45`): `LoginResponse.tokens` is non-null when `accessToken != null && refreshToken != null`, and `mfaRequired` is `tokens == null && mfaToken != null`. Empty strings are non-null, so `tokens = TokenResponse("", "")`, `mfaRequired = false`.
- Consequence (verified, `app/src/main/java/gg/refx/android/data/repo/AuthRepository.kt:33-46,56-61`): `login()` takes the success branch, persists empty tokens, calls `GET auth/me` with `Authorization: Bearer ` â†’ 401 â†’ TokenAuthenticator refreshes with an empty refresh token â†’ sign-out. An MFA-enabled user cannot sign in on Android.
- Verdict: panel report correctly states the wire shape (its Â§2 says `accessToken: ""`); the Android report's phrase "mfaRequired when tokens null" accurately describes the client code but that client assumption is wrong against this backend. **Desktop rule: detect the MFA branch via `mfaRequired == true` or non-blank `mfaToken`, never via null tokens.**

**A2. Variable update: method+path disagreement â€” panel report is right about the server contract.**
- Android (verified, `app/src/main/java/gg/refx/android/data/api/ServerSettingsApi.kt:21-22`): `@PATCH("servers/{id}/variables")` with body `UpdateVariableRequest(envName, value)` (`app/src/main/java/gg/refx/android/data/model/ServerSettings.kt:31-34`; call site `data/repo/ServerManageRepositories2.kt:38-39`).
- Panel (verified, `apps/panel-api/src/servers/servers.controller.ts:733-766`): only `GET :serverId/variables`, `PUT :serverId/variables/:envName` (body `SetVariableDto { value }`), `DELETE :serverId/variables/:envName`. No `PATCH .../variables` route exists anywhere (checked the full `@Patch` inventory of the controller: `:id/startup`, `:id/auto-restart`, `:id/minecraft-version`, `:id/minecraft`, `:id/workshop/reorder`, `:id/workshop/:modId`, `:serverId/resize`, `:serverId/sub-users/:subUserId`, `:serverId/schedules/:scheduleId`).
- Consequence: Android's variable update 404s. Even if the route existed, `forbidNonWhitelisted` would 400 the extra `envName` body field. **Desktop rule: `PUT /api/v1/servers/:serverId/variables/:envName` with body `{ value }`.**

**A3. API-key mint plaintext field: Android decodes `key` from the endpoint that returns `token` â€” panel report is right.**
- Android (verified): calls `POST account/api-keys` (`app/src/main/java/gg/refx/android/data/api/AccountApi.kt:85-86`) decoding `CreatedApiKey(val key: String, val prefix: String, val id: String)` with `key` required/non-defaulted (`app/src/main/java/gg/refx/android/data/model/AccountExtras.kt:57-66`, doc comment claims "`{ key, prefix, id }`").
- Panel (verified): `POST /api/v1/account/api-keys` returns `{ id, name, prefix, scopes, expiresAt, token: plaintext }` â€” `apps/panel-api/src/account/account.controller.ts:105-115`, including the verbatim comment "Field name MUST match the web client's ApiKey.token (the copy-once UI)". The `key` field exists only on `POST /api/v1/auth/api-keys`, which returns `{ key: plaintext, prefix, id, expiresAt }` (`apps/panel-api/src/auth/auth.controller.ts:310-328`).
- Consequence: kotlinx decode fails on the missing required `key` (ignoreUnknownKeys doesn't supply missing fields) â†’ the Android create-key feature errors with a generic message. The Android fixture in `AccountSecurityDecodingTest.kt` ("Server returns exactly { key, prefix, id }") is wrong for the endpoint the app calls; its `rfx_ab12...` prefix is also cosmetically wrong vs the real `refx_` format. **Desktop rule: read `token` from `/account/api-keys` or `key` from `/auth/api-keys` â€” they differ by design.**

**A4. Signed download URLs: panel mints relative paths; Android's opener is https-only â€” file downloads always fail on Android.**
- Panel (verified): `GET .../files/download-url` returns `{ url: "/servers/<id>/files/download?path=...&exp=...&sig=..." }` â€” a **relative** path with no `/api/v1` prefix (`apps/panel-api/src/files/files.service.ts:135-149`). Backups: `.../backups/:backupId/download` returns an absolute S3 presigned URL when the agent provides one (`if (url && /^https?:\/\//i.test(url)) return { url }`), else the relative relay `/servers/:id/backups/:backupId/archive?exp&sig` (`apps/panel-api/src/backups/backups.service.ts:270-300`).
- Android (verified): repos pass `.url` through raw (`data/repo/ServerManageRepositories.kt:28,37`); the screens hand it to `WebLink.open`, which parses the URI and returns false unless `uri.scheme` is `https` (`app/src/main/java/gg/refx/android/core/ui/WebLink.kt:12-15`; `feature/servers/sections/FilesScreen.kt:189-194`). A relative URL has a null scheme â†’ "Couldn't open the download link."
- Consequence: Android file downloads are always broken against this backend; backup downloads work only in the S3-presigned case. **Desktop rule: if the returned `url` does not match `^https?://`, resolve it as `apiOrigin + "/api/v1" + url` (both the `/api/v1` prefix and origin are missing from the minted string).**

### B. Report-level disagreements and omissions

**B1. Error envelope: the Android report's `{ success:false, error:{ message, code } }` shape is never emitted by this backend â€” panel report is right.**
- Verified: the only error producer is `AllExceptionsFilter` emitting `{ statusCode, error: <string code>, message: string|string[], path, timestamp }` (`apps/panel-api/src/common/filters/all-exceptions.filter.ts:42-48`); the success interceptor only ever emits `success: true` (`apps/panel-api/src/common/interceptors/transform.interceptor.ts:42-51`).
- Consequence for Android (verified `core/network/ApiCall.kt:27-43`): on non-2xx it decodes the body as `ApiEnvelope<JsonElement>` and reads `.error` â€” but the real `error` field is a JSON **string**, so decoding into `ApiErrorBody?` throws, `runCatching` yields null, and the user always sees the per-status fallback ("You don't have permission to do that.", etc.). Real server messages (e.g. "Missing permissions: â€¦", "Backup is locked â€” unlock it first") never surface. The `success:false` branch in `EnvelopeConverterFactory.kt:62-70` is dead code against this backend (non-2xx bypasses the converter anyway).
- **Desktop rule: parse errors as `{ statusCode, error, message (string OR string[]), path, timestamp }`.**

**B2. Support routes: Android's `PATCH support/tickets/{id}` and `POST support/tickets/{id}/assign` exist but are staff-only; panel report omitted them.**
- Verified `apps/panel-api/src/support/support.controller.ts`: `@Patch("tickets/:id")` (line 83) and `@Post("tickets/:id/assign")` (line 125) are both `@UseGuards(AdminPermissionGuard)` + `@RequirePerm("support.manage")`. Bonus routes both reports missed: `GET support/staff`, `POST support/tickets/:id/archive`, `DELETE support/tickets/:id`, canned-responses CRUD, `GET support/kb-articles` (+staff CRUD) â€” same controller.

## (c) Confidence verdicts on the three perilous contracts

1. **Envelope convention â€” HIGH confidence, with a split verdict.** Success: `{ success: true, data: <T> }`, paginated `{ success: true, data: [...], meta: { page, pageSize, total, totalPages } }`, null/undefined handler returns pass through unwrapped (204s bodyless) â€” both reports agree and source confirms (`transform.interceptor.ts:42-51`). Errors: **only** the flat `{ statusCode, error: <string>, message: string | string[], path, timestamp }` shape exists (`all-exceptions.filter.ts:42-48`); the `{ success:false, error:{ message, code } }` shape in the Android client is dead code against this backend. A desktop client that copies Android's error parsing will show generic messages forever; parse the flat shape.
2. **Auth/refresh flow â€” HIGH confidence.** `POST /api/v1/auth/refresh`, body `{ refreshToken }`, no Authorization header required, response `{ accessToken, refreshToken, expiresIn }`, rotating with family-revocation-on-reuse; refresh-once-then-retry on 401 with a bare (interceptor-free) HTTP client is the correct pattern (TokenRefresher/TokenAuthenticator verified). **One landmine: the MFA login branch (A1)** â€” detect it via `mfaRequired`/`mfaToken`, not null tokens, because the panel sends `accessToken: ""`.
3. **Console handshake â€” HIGH confidence.** Socket.IO at the API origin, engine path default `/socket.io`, namespace `/ws/console`, authenticated with the ordinary access JWT in `handshake.auth.token` (Bearer header also accepted); no console-token REST endpoint exists in either repo (both reports agree; gateway source confirms). `subscribe { serverId }` â†’ `subscribed`; `command { command }` requires the literal `"control.console"` permission for sub-users (unreachable via the sub-user grant DTO â€” owners unaffected; sub-users must use REST `POST /servers/:id/command`). Extra guard found in source: connections are rejected for non-ACTIVE or `mustChangePassword` accounts (`console.gateway.ts:59-65`).
