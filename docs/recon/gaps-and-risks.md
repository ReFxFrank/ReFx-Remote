# Completeness critic: desktop-critical gaps (spot-checked)

> Recon agent output, 2026-07-13. Source-grounded; file paths cited are relative to the named repo.

## Summary

The recon is strong on REST surface and WS protocol but leaves seven desktop-critical gaps. I spot-checked the top three in source and confirmed: (1) API keys (WRITE/ADMIN scope, X-Api-Key header) can drive every REST mutation including power/command because they inherit the owner's full RBAC — but the /ws/console Socket.IO handshake verifies ONLY an access JWT (jwt.verifyAsync against accessSecret), so an API-key-only desktop app has no live console/stats/power feed; a long-lived desktop session therefore must use the JWT login+refresh flow, whose rotation mints a brand-new session with a full TTL every refresh (30d default, 90d with rememberMe) — effectively sliding, so only a device idle longer than the TTL gets signed out, but a crash between rotation and token persistence outside the 60s grace window revokes the whole session family. (2) Desktop crash detection has no usable push primitive: push-tokens DTO hard-rejects any platform except 'ios'/'android' and delivery only queries platform:'ios' (APNs) — the desktop's options are per-server WS 'power' events (requires a live subscribed socket; commands on a shared socket target only the LAST subscribed server, and there is no unsubscribe event) or polling GET /account/notifications (in-app notices exist only for CRASHED/SUSPENDED, owner-only, throttled 30 min per server+state). (3) File upload is capped hard at 32 MiB through the panel with the server explicitly telling clients "Use SFTP for larger files"; the upload-url endpoint is unusable by clients (the node agent returns a relative agent-API path guarded by node-HMAC, with a TODO to mint real signed tokens), and there is no chunked/multipart/resumable upload — downloads however ARE resumable (Range passthrough) via a signed relative URL that the client must prefix with <origin>/api/v1 itself.

## Gap 1 (SPOT-CHECKED): Desktop auth architecture — API keys vs JWT are two disjoint capability sets

The recon documents both credential types but never states the decisive fact: **which one a desktop app must use, and why it may need BOTH.**

**Verified from source:**

- `apps/panel-api/src/auth/guards/jwt-auth.guard.ts:42-49` — API keys are accepted only via the `x-api-key` header; `req.user` is populated by `ApiKeyService.authenticate` and the request proceeds without ever touching JWT verification.
- `apps/panel-api/src/auth/api-key.service.ts:119-126` — an authenticated API key returns an `AuthUser` with `permissions: permissionsForGlobalRole(key.user.globalRole)` plus `apiKeyId`/`apiKeyScopes`. The key impersonates its owner for RBAC purposes.
- `apps/panel-api/src/common/interceptors/api-key-write-scope.interceptor.ts:51-78` — the only ceiling: keys carrying `WRITE` or `ADMIN` scope may issue POST/PUT/PATCH/DELETE; `READ`-only keys are limited to safe methods; `STATUS_READ`-only keys are rejected on every JwtAuthGuard route with 403 `'This token is scoped to the status feed only'`. **Conclusion: a WRITE-scoped `refx_` key CAN do power, command, file writes, backups — the full REST control surface.**
- `apps/panel-api/src/agent/console.gateway.ts:44-72` — the WS handshake reads `handshake.auth.token` or the `Authorization: Bearer` header and calls `this.jwt.verifyAsync(token, { secret: jwtCfg.accessSecret })`. There is no `ApiKeyService` path here. **A `refx_` key can never open the console socket.** An API-key-only desktop app is REST-only: no live console, no live stats frames, no push power events.

**Refresh-token idle semantics (verified):**

- `apps/panel-api/src/auth/auth.service.ts:33-43` — `TRUSTED_REFRESH_TTL_SECONDS = 90 * 24 * 60 * 60` with the comment that trusted ("remember me") lifetime is "Sliding (re-extended on every refresh), so an active device effectively never has to re-login."
- `auth.service.ts:325-356` (`issueTokens`) — every rotation creates a NEW session row with `expiresAt: new Date(Date.now() + refreshTtl * 1000)`, where `refreshTtl = trusted ? 90d : jwtCfg.refreshTtl (30d default)`. So the non-remembered 30d TTL is **also** re-extended on every refresh — both modes are sliding-on-use. A desktop user is signed out only after being fully idle (no refresh) for 30/90 days.
- `auth.service.ts:359-426` — rotation semantics a desktop client must survive: presenting an already-rotated token is tolerated only within `REFRESH_ROTATION_GRACE_MS` (60s per recon) of `rotatedAt`; outside that window it calls `revokeFamily(userId)` (`auth.service.ts:441-446`) which revokes **every active session for the user** — including their browser and phone. Design consequence: the desktop app must persist the rotated refresh token atomically (write-before-use) and single-flight all refreshes, or a crash mid-rotation can log the user out of every device.

**What the designer should decide:** JWT login (+ MFA challenge) is mandatory for the realtime experience; API keys are only viable for a headless/tray "monitor mode." No device flow, OAuth, or device pairing exists anywhere (confirmed by recon 4 and my grep — no client-version or device endpoints in `apps/panel-api/src`).

## Gap 2 (SPOT-CHECKED): Crash detection / notifications — no desktop push primitive exists

**Verified from source:**

- `apps/panel-api/src/account/dto/push-token.dto.ts:11-13` — `@IsIn(['ios', 'android'])` on `platform`. `POST /account/push-tokens { token, platform: 'windows' }` would 400. There is no desktop registration path at all.
- `apps/panel-api/src/push/push.service.ts:85-104` — `sendToUser` queries `pushToken.findMany({ where: { userId, platform: 'ios' } })` and delivers via APNs only. Even registered `'android'` tokens are never delivered (matches docs/16-status.md "FCM TODO(impl)"). **Adding a `'windows'` platform value would still deliver nothing.**
- `apps/panel-api/src/agent/agent-callbacks.controller.ts:35-50` — two disjoint notice tables: in-app `SERVER_STATE_NOTICES` = `{ CRASHED: 'has crashed', SUSPENDED: 'was suspended' }` only; push `SERVER_STATE_PUSH` = `{ RUNNING, OFFLINE, CRASHED }`.
- `agent-callbacks.controller.ts:365-445` (`applyServerState`) — on a state transition it (a) persists state, (b) creates an in-app notification for the **owner only** (`server.ownerId` — sub-users get nothing) throttled 30 min per server+state (`STATE_NOTICE_THROTTLE_MS`, line 55), (c) sends push with `type: 'server.state', data: { serverId }`. Unchanged state = no-op.
- Notification REST surface (verified routes, `apps/panel-api/src/account/account.controller.ts:183-225`): `GET /account/notifications` (plain array, not paginated envelope — "The web expects a plain array" comment at line 189), `GET /account/notifications/unread-count`, `POST /account/notifications/:id/read`, `POST /account/notifications/read-all`, `DELETE /account/notifications/:id`, `DELETE /account/notifications` (clear all).

**Desktop crash-detection options, in order of latency:** (1) hold a Socket.IO connection and subscribe to each watched server — `power` events arrive in real time; (2) poll `GET /account/notifications/unread-count` (only CRASHED/SUSPENDED, owner-only, 30-min throttle means a crash-loop shows once); (3) poll `GET /servers/:id/stats` or `GET /servers` for state. There is NO account-wide WS event stream — the gateway only emits into per-server rooms (`console.gateway.ts:122-135`).

**Bonus WS quirk verified (not in any recon):** `console.gateway.ts:94` sets `client.data.serverId = body.serverId` — a single scalar overwritten by each `subscribe`. Rooms accumulate (multi-server events on one socket work), but `command` (`console.gateway.ts:99-118`) always targets the **last-subscribed** server, and **no `unsubscribe` handler exists** (only `subscribe` and `command`). A multi-server desktop dashboard should use one socket per server for command routing, or emit `subscribe` immediately before every `command`.

## Gap 3 (SPOT-CHECKED): File transfer mechanics — 32 MiB hard ceiling, upload-url is a dead end, SFTP is the official big-file path

**Verified from source:**

- `apps/panel-api/src/files/files.service.ts:30-31` — `MAX_UPLOAD_BYTES = 32 * 1024 * 1024` ("Matches the node agent's signed-body cap (32 MiB) on /files/write"); line 204 rejects with `'File exceeds the 32 MiB direct-upload limit. Use SFTP for larger files.'`
- `apps/panel-api/src/main.ts:114-115` — `POST /api/v1/servers/:id/files/upload` uses `raw({ type: () => true, limit: "40mb" })`; lines ~206-209 note a 240s `requestTimeout` covering the whole request body (web client caps uploads at 180s).
- **upload-url is unusable by any client:** `apps/panel-api/src/files/files.service.ts:184-186` proxies to `agent.fileUploadUrl` (`apps/panel-api/src/agent/agent.client.ts:633-643`), and the agent handler `apps/node-agent/internal/api/handlers_files.go:304-314` returns `{"url": "/api/v1/servers/<id>/files/write?path=..."}` — a **relative URL into the agent's own HMAC-node-signed API**, with the comment `TODO(impl): mint a signed, single-use upload token as above`. A desktop client cannot sign agent requests. Do not design against `POST .../files/upload-url`.
- No chunked, multipart, or resumable upload exists anywhere in `files.controller.ts` / `handlers_files.go`. For >32 MiB (modpacks, world uploads) the only path is SFTP on the node (port 2022; `GET /servers/:id/sftp` → `{host, port, username}`, `POST /servers/:id/sftp/rotate` → one-time password). A Tauri app needs a Rust SFTP client (e.g. ssh2/russh) — this is an app-architecture requirement no recon surfaced.
- **Downloads ARE resumable:** `apps/panel-api/src/files/files.controller.ts:104-132` — the `@Public()` `GET .../files/download` verifies the HMAC (`exp`/`sig` query params) and passes through `Range`/`Content-Range`/`Content-Length` ("resumable downloads with real progress").
- **Relative-URL trap resolved:** `files.service.ts:135-149` — `downloadUrl` returns `url: '/servers/<id>/files/download?path=..&exp=..&sig=..'` **without** the `/api/v1` prefix. Since the route lives under the global prefix, the client must prepend `<apiOrigin>/api/v1` itself. TTL is `DOWNLOAD_TTL_SECONDS` (300s per recon, declared line 33-34 area). Backup downloads have the same relative-vs-S3-absolute duality (recon 1).

## Remaining gaps (not spot-checked) — where the answers live

**Gap 4 — Rate-limit budget vs desktop polling patterns.** Global 120 req/60s **per IP** (`apps/panel-api/src/app.module.ts:83-101`). A desktop dashboard polling `GET /servers/:id/stats` every 5s for N servers burns N×12 req/min — 10 servers saturates the entire budget, starving auth refreshes. Mitigation is the WS stats feed (no HTTP throttle applies; agent callbacks are `@SkipThrottle`). The exact 429 headers (`Retry-After` vs `X-RateLimit-*`) are decided by `@nestjs/throttler` v6 defaults — answer lives in `node_modules/@nestjs/throttler/dist/throttler.guard.js` (`ThrottlerGuard.throwThrottlingException` / header-setting code), or one live 429 against a dev stack.

**Gap 5 — Session revocation UX.** The desktop session is visible/revocable from web (`GET/DELETE /account/sessions` — `apps/panel-api/src/account/account.controller.ts`, backing query `auth.service.ts:1034-1041` filters `revokedAt: null, expiresAt > now`). Desktop must treat refresh-401 as "signed out remotely," not an error loop. Also unverified: whether the Session row's `userAgent` would render usefully for a Tauri/reqwest client — the desktop should send a distinctive User-Agent so users can recognize it in the sessions list. Answer: `auth.controller.ts:142,166,177` (userAgent capture) + Session model in `database/prisma/schema.prisma`.

**Gap 6 — App update/versioning.** Grep across `apps/panel-api/src` found no minimum-client-version, version-negotiation, or User-Agent gating (only outbound User-Agent strings and audit-log capture). The GitHub-Releases self-update pattern exists only for the node agent (`apps/node-agent/internal/api/handlers_system.go:20`, repo `refxfrank/refxhosting`). A Tauri updater needs its own release channel + update manifest; if the owner wants server-driven "please update" messaging, the natural home is a new public endpoint beside `apps/panel-api/src/status/status.controller.ts` — nothing exists today.

**Gap 7 — Notification row wire shape.** `GET /account/notifications` returns a plain array of Notification rows, but no recon extracted the field list. Answer: `apps/panel-api/src/platform/notifications.service.ts` (`createNotification` writes `title`, `body`; `listNotifications` select) + the `Notification` model in `database/prisma/schema.prisma`.

**Gap 8 — Reconnect/resync procedure.** No scrollback and no missed-event replay on the WS: after reconnect the desktop must re-`subscribe`, then resync via `GET /servers/:id` (state) + `GET /servers/:id/stats`, and refetch backups/schedules if those views are open. Whether the gateway's `canAccess` (`console.gateway.ts:139+`) blocks SUSPENDED servers from subscribing (the HTTP PermissionGuard does — `permission.guard.ts:99-109`) was not read — check `console.gateway.ts` `canAccess` body before assuming WS parity with the suspended-server 403.

**Gap 9 — SFTP credential model for multi-device use.** `POST /servers/:id/sftp/rotate` shows a password once; if the password is per-server (not per-user/per-device), a desktop rotating it silently breaks the user's saved FileZilla/other-device credentials. Whether it is per-server or per-user, and whether sub-users share it, lives in `apps/panel-api/src/sftp/` and the agent's SFTP auth in `apps/node-agent/internal/sftp/`.

**Gap 10 — Socket.IO client for Rust.** Recon establishes Socket.IO v4, namespace `/ws/console`, default engine path `/socket.io`, transports websocket+polling, `cors: { origin: true }` (no Origin problem for Tauri). The design must pick a Rust client (`rust-socketio` supports EIO=4 but its reconnect/backoff and auth-payload ergonomics differ from the JS client) or run the socket in the Tauri webview with the JS client and bridge to Rust. Nothing in any repo answers this; it is a pure desktop-side decision, but the handshake contract to test against is fully specified in `apps/panel-api/src/agent/console.gateway.ts:44-72`.
